# Prototype Sketching Guide

**Time in lab:** 10 minutes (start; continue after lab)  
**Purpose:** Translate your validated problem into a tangible first concept that can be tested with users in Lab 5.

---

## What You Are Building

A low-fidelity prototype: rough, quick, and disposable. This is NOT your final product. This is a conversation tool that lets you test your assumptions with real users next week.

**Low-fidelity means:**
- Paper sketches with marker on printer paper
- Figma wireframes using rectangles and text (no colors, no branding)
- A series of 4 to 6 screens showing the core user flow
- Rough enough that people feel comfortable criticizing it

**Low-fidelity does NOT mean:**
- A polished mockup with gradients and shadows
- A coded prototype
- Anything that took more than 2 hours to create

---

## The Core Flow Method

### Step 1: Identify the Critical Path (3 minutes)

Based on your problem statement, what is the one thing your user needs to accomplish? Map it as a series of steps:

1. User starts here: [Entry point: how do they find your product?]
2. User does this: [First action they take]
3. User sees this: [The key information or value moment]
4. User achieves this: [The outcome that solves their problem]

This is your critical path. You will sketch one screen for each step.

### Step 2: Sketch Each Screen (5 minutes)

For each step in the critical path, draw a rectangle (phone screen or browser window) and sketch:

- What the user sees (main content area)
- What the user can do (buttons, inputs, navigation)
- What information is displayed
- How they get to the next screen (arrow or annotation)

**Sketching tips:**
- Use thick markers, not pens. This forces you to stay rough.
- Write annotations outside the screen: "tap here to see list" or "shows real-time data"
- Do not worry about pixel precision. Worry about whether the flow makes sense.
- Number your screens: Screen 1, Screen 2, Screen 3, Screen 4.

### Step 3: Label the Assumptions (2 minutes)

On each screen, write down one assumption you are making. For example:

- Screen 1: "We assume users will find us through a QR code on campus"
- Screen 2: "We assume users will enter their current building"
- Screen 3: "We assume real-time data will be accurate enough to be useful"
- Screen 4: "We assume users will check in when they arrive"

These assumptions become your testing questions for Lab 5 user testing.

## If Using Figma

1. Create a new Figma file: `[Team Name] - Prototype v1`
2. Use only rectangles, text, and lines. No components, no design system.
3. Create one frame per screen (use iPhone 14 or Desktop 1440 frame size)
4. Connect screens with Figma's prototype mode (click, then drag to next frame)
5. Export each screen as PNG and save to your repo at `/02-design/prototypes/low-fidelity/`

## This Is Not Graded Separately

The prototype is part of Checkpoint 1 and will be tested with users in Lab 5. The quality bar is low: it just needs to be clear enough that a stranger can understand the flow and give you feedback. You will iterate on it significantly in the coming weeks.
